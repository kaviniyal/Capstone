B
Query Pipeline Flow — Every Investigation
1
🛡️ User Input → Guardrails
Analyst types query in React frontend → HTTP POST to FastAPI → Guardrails validate
INPUT: "Sedan collision, third party fault, police report filed" GUARDRAILS: Length check → 56 chars ≤ 2000 ✓ Injection scan → no attack patterns ✓ PII scan → no personal data found ✓ OUTPUT: sanitized_query = "Sedan collision, third party fault, police report filed" guardrail_flags = ["OK"] blocked = False
2
🔍 Retrieval — BM25 + Pinecone → RRF → Cross-Encoder → CRAG
Sanitized query runs through 4-stage hybrid retrieval to find top 5 similar historical claims
BM25 SEARCH (docs_cache.json): query tokens: ["sedan", "collision", "third", "party", "police", "report"] → scored 15,420 docs → top 15 by keyword match PINECONE SEARCH: embed query → [0.019, -0.401, 0.854, ...] 1536 dims cosine similarity against 15,420 stored vectors → top 15 by semantic similarity RRF FUSION: doc_A: BM25 rank=1 → 1/61=0.0164 | Pinecone rank=2 → 1/62=0.0161 | total=0.0325 doc_B: BM25 rank=2 → 1/62=0.0161 | Pinecone rank=1 → 1/61=0.0164 | total=0.0325 → merged top 20 by combined rank score CROSS-ENCODER (ms-marco-MiniLM-L-6-v2): ["sedan collision third party police report" + doc_1] → score: 4.87 ✓ ["sedan collision third party police report" + doc_2] → score: 3.21 ✓ → top 5 by joint relevance score CRAG CHECK: avg_score = 3.21 > 0.30 threshold → results OK → crag_triggered = False OUTPUT: retrieved_claims = [5 documents with metadata + scores]
3
🔗 Correlation Agent — Pattern Detection
Analyses the 5 retrieved claims together for suspicious cross-claim patterns
STATISTICAL PRE-FILTER (Python, no LLM): regions: ["Urban","Urban","Rural","Urban","Urban"] → Urban 4/5 → region_hotspot fraud: ["Y","Y","N","N","Y"] → 3/5 = 60% → high_fraud_density amounts: [18000,17500,19000,18500,20000] → CV=0.05 → amount_cluster LLM ANALYSIS (GPT-4o-mini): input: query + 5 claim documents + statistical signals output: { signals: [REGION_HOTSPOT(HIGH), AMOUNT_CLUSTER(MEDIUM)], overall_correlation_risk: "HIGH", summary: "Urban sedan collision pattern with similar amounts...", investigation_flags: ["Cross-check claims 1 & 3 for staging"] } OUTPUT: correlation_signals → state
4
📊 Risk Scoring Agent — Fraud Probability
GPT-4o-mini reads legitimacy signals from query + retrieved claims → outputs fraud probability
LEGITIMACY SIGNALS in query: "police report filed" → -0.3 "third party fault" → -0.25 total adjustment → -0.55 LLM SCORING (GPT-4o-mini): input: query + 5 historical claims output: { fraud_probability: 0.22, risk_level: "LOW", confidence: 0.81, key_risk_factors: ["Third party at fault is a legitimacy indicator", "Police report filed — standard procedure followed"], requires_human_review: False ← 0.22 not in HITL band [0.4, 0.6] } A2A MESSAGE POSTED: APPROVE → recommendation_agent (fraud_prob=0.22 < 0.4) OUTPUT: risk_assessment + a2a_messages → state
5
📋 Policy Validation Agent — Compliance Check
Checks ONLY what is explicitly stated in the query against policy rules
EXPLICIT SIGNALS READ: "police report filed" → police report IS filed (positive) "third party fault" → not policy holder fault (positive) no mention of "no police report" → no violation LLM VALIDATION (GPT-4o-mini): output: { is_policy_valid: True, violations: [], eligibility_flags: [], validation_summary: "Claim appears to comply with policy terms. Third party fault and filed police report are positive legitimacy indicators." } A2A MESSAGE: None (no violations) OUTPUT: policy_validation → state
6
👤 HITL Gate — Human Review Check
Pure Python check — no LLM call. Checks if fraud probability is in uncertainty band.
fraud_probability = 0.22 HITL_LOW = 0.40 HITL_HIGH = 0.60 0.22 between 0.40 and 0.60? → NO → awaiting_human = False ROUTING: → recommendation node (continue automatically) OUTPUT: awaiting_human = False → state
7
✅ Recommendation Agent — Final Decision
Reads ALL upstream data + A2A messages → GPT synthesises final recommendation
ALL INPUTS READ: human_decision: "" (no human override) fraud_probability: 0.22 (LOW) risk_level: "LOW" policy_valid: True violations: [] correlation_risk: "HIGH" (2 signals) a2a_messages: [APPROVE from risk_scoring_agent] LLM SYNTHESIS (GPT-4o-mini): Human decision: None → use AI rules fraud_prob 0.22 < 0.3 AND no violations → APPROVE But correlation risk is HIGH → add investigation note output: { decision: "APPROVE", priority: "P4", investigation_steps: [ "Standard document verification", "Monitor for related claims in same Urban cluster" ], evidence_summary: "Low fraud probability (22%) with positive legitimacy indicators — third party fault, police report filed, no past claims...", estimated_fraud_savings: "N/A — claim approved" } OUTPUT: recommendation → state
8
💾 Save to Pinecone + Return Response
Investigation saved permanently to Pinecone as a new vector. FastAPI returns JSON to React frontend.
SAVE TO PINECONE: embed: "Fraud probability 0.22. LOW risk. Decision: APPROVE. Third party fault..." → vector: [0.041, -0.312, ...] 1536 dims → upsert: id=investigation_a3f2c1d4 metadata: {record_type: "investigation", decision: "APPROVE", fraud_probability: "0.22", timestamp: "2025-06-03 10:45 UTC"} API RESPONSE (JSON → React frontend): { "thread_id": "abc-123", "awaiting_human": false, "risk_assessment": {"fraud_probability": 0.22, "risk_level": "LOW"}, "policy_validation": {"is_policy_valid": true, "violations": []}, "correlation_signals": {"overall_correlation_risk": "HIGH"}, "recommendation": {"decision": "APPROVE", "priority": "P4"} } FRONTEND: renders Risk bar (22%), Policy (✓ Valid), Recommendation (APPROVE P4) HISTORY TAB: investigation appears in list
