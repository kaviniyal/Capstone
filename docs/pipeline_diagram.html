01
🛡️
Guardrails — Input Validation
Safety gate — validates and sanitizes every query before it touches any AI agent
▼
📏
Length Check
Query must be ≤ 2000 characters. If longer — truncate and add TRUNCATED flag. Prevents token overflow in LLM context window.
🚫
Prompt Injection Detection
Scans for patterns like "ignore previous instructions", "act as", "jailbreak", "DAN". If found → BLOCKED, pipeline stops immediately.
ignore previous instructions act as jailbreak
🔐
PII Redaction
Layer 1 — Microsoft Presidio (NLP): Detects PERSON, EMAIL, PHONE, SSN, CREDIT_CARD using spaCy en_core_web_lg model.
Layer 2 — Regex fallback: Catches email, phone, SSN, card patterns if Presidio not installed.
Replaces PII with [EMAIL], [PHONE], [SSN], [CARD] placeholders.
Output → State
sanitized_query guardrail_flags: ["OK" | "TRUNCATED" | "INJECTION_DETECTED" | "PII_*"] blocked: true/false
sanitized query
02
🔍
Hybrid Retrieval — Find Similar Claims
BM25 + Pinecone + RRF Fusion + Cross-Encoder Reranking + CRAG quality check
▼
🔤
BM25 — Keyword Search
Runs on docs_cache.json (15,420 documents in memory). Scores documents by word frequency using BM25Okapi. Returns top 15 by exact keyword match. Fast, no API call, finds specific terms like policy numbers.
rank_bm25 local · no API
🧠
Pinecone — Semantic (Meaning) Search
Embeds query using OpenAI text-embedding-3-small (1536 dimensions). Queries Pinecone cloud vector database using cosine similarity. Returns top 15 by meaning — finds claims with same concept even if different words.
text-embedding-3-small 1536 dimensions cosine similarity
⚖️
RRF Fusion — Combine Both Lists
Reciprocal Rank Fusion merges BM25 (15) + Pinecone (15) results. Score = 1/(60+rank) for each list. No score normalisation needed — rank-based. Documents appearing in both lists score highest. Returns top 20 combined.
1/(60+rank) rank-based
🎯
Cross-Encoder Reranking
ms-marco-MiniLM-L-6-v2 reads query + each of 20 candidates TOGETHER. Gives precise relevance score. More accurate than bi-encoder because model sees both at once. Returns top 5 most relevant claims.
ms-marco-MiniLM-L-6-v2 local · CPU
🔄
CRAG — Quality Check
Corrective RAG checks average cross-encoder score of top 5. If avg < 0.30 → results not confident enough → LLM rewrites query with more specific insurance terms → retry all 4 steps. Sets crag_triggered=True if retry needed.
threshold: 0.30 auto-retry
Output → State
retrieved_claims [5 documents with scores] query_used crag_triggered: true/false
5 similar claims
03
🔗
Correlation Agent — Cross-Claim Pattern Detection
Analyses the 5 retrieved claims together to find suspicious patterns across them
▼
📊
Phase 1 — Statistical Pre-Filter (Pure Python · No LLM)
Fast rule-based checks before any LLM call:
• Region Hotspot: Count customer_region — flag if any region ≥ 2 occurrences
• Fraud Density: Flag if >60% of 5 claims are labelled fraud
• Amount Clustering: Coefficient of Variation <0.12 → suspiciously similar amounts
• Repeat Customer: Same policy number appearing more than once
🤖
Phase 2 — LLM Analysis (GPT-4o-mini)
Statistical signals injected into prompt. GPT-4o-mini analyses all 5 claims together and identifies named fraud patterns with severity levels.
REPEAT_CUSTOMER REGION_HOTSPOT AMOUNT_CLUSTER TEMPORAL_BURST PATTERN_MATCH
Output → State
correlation_signals.signals [] overall_correlation_risk: LOW/MEDIUM/HIGH/CRITICAL summary investigation_flags [] statistical_pre_signals {}
correlation signals
04
📊
Risk Scoring Agent — Fraud Probability
Estimates fraud probability 0–1 using GPT-4o-mini and posts A2A messages
▼
🤖
LLM Fraud Probability Scoring
GPT-4o-mini reads the query + top 5 retrieved claims. Estimates fraud probability based on how many retrieved claims are fraudulent and what patterns they share with the query.
0.0–0.3 → LOW 0.3–0.5 → MEDIUM 0.5–0.75 → HIGH 0.75–1.0 → CRITICAL
⚡
HITL Band Check
Sets requires_human_review=True if probability falls in uncertainty band [0.4, 0.6]. This flag will be read by Node 6 to decide whether to pause the pipeline.
band: 0.4 → 0.6
📨 A2A Messages Posted → recommendation_agent
ESCALATE If fraud_prob > 0.6 → "High fraud probability detected: {prob}"
APPROVE If fraud_prob < 0.4 → "Low risk — safe to approve"
FLAG If correlation_risk = HIGH/CRITICAL → "CORRELATION_{level}"
Output → State
risk_assessment.fraud_probability risk_level: LOW/MEDIUM/HIGH/CRITICAL confidence key_risk_factors [] requires_human_review a2a_messages +ESCALATE/APPROVE/FLAG
fraud score + A2A
05
📋
Policy Validation Agent — Compliance Check
Checks the claim against insurance policy rules — finds explicit violations only
▼
📜
Compliance Rules Check (GPT-4o-mini)
Checks ONLY what is EXPLICITLY stated in the query:
• Police report explicitly NOT filed → VIOLATION
• Claim filed immediately after policy start → VIOLATION
• Claim amount exceeds vehicle value → VIOLATION
• Multiple recent claims explicitly mentioned → VIOLATION

Key rule: "police report" alone means FILED. Only flag if query says "no police report" or "not filed".
📨 A2A Message Posted → recommendation_agent
FLAG If violations found → "POLICY_VIOLATION: {detail}"
Output → State
policy_validation.is_policy_valid violations [] eligibility_flags [] validation_summary a2a_messages +FLAG
violations + A2A
06
👤
HITL Check — Human-in-the-Loop Gate
Pauses the pipeline if fraud score is in the uncertainty band (0.4–0.6)
▼
🔢
Pure Python Logic — No LLM Call
Reads fraud_probability from Node 4. Checks if it falls between HITL_LOW (0.4) and HITL_HIGH (0.6) configured in .env. Simple boolean check — fastest node in the pipeline.

07
✅
Recommendation Agent — Final Decision
Reads ALL upstream outputs + A2A messages and generates the final investigation decision
▼
📥
Reads All Inputs
Receives everything accumulated from previous nodes:
• Original query
• Risk assessment (prob, level, factors) from Node 4
• Policy violations from Node 5
• Correlation signals (pattern type, severity) from Node 3
• All A2A messages (ESCALATE/FLAG/APPROVE) from Nodes 4+5
• Human decision (if HITL triggered) from Node 6
• Top 3 similar historical claims from Node 2
🤖
LLM Synthesis (GPT-4o-mini)
GPT reads everything together and applies decision rules: APPROVE if prob <0.3 and no violations. INVESTIGATE if 0.3–0.6 or minor violations. ESCALATE if prob >0.6 or ESCALATE A2A received. REJECT if clear policy ineligibility.
APPROVE → P4 INVESTIGATE → P2/P3 ESCALATE → P1/P2 REJECT → P3
Output → State → Saved to Pinecone → Returned to Frontend
decision: APPROVE/INVESTIGATE/ESCALATE/REJECT priority: P1/P2/P3/P4 investigation_steps [] evidence_summary escalation_reason estimated_fraud_savings
save to Pinecone + return JSON

Pipeline Orchestration — LangGraph ✅ Chosen
Controls the flow between all 7 nodes with HITL checkpointing
▼
Option HITL Support Deterministic Complexity Reason
✅ LangGraph First-class Yes Medium HITL is a first-class feature. Deterministic flow. Full state history. MemorySaver checkpointing.
❌ LangChain AgentExecutor None No Low LLM decides tool order — non-deterministic. No HITL support. Hard to audit.
❌ Custom Python Manual Yes High Would need to rebuild checkpointing, state management, routing — reinventing LangGraph

CH
📄
Chunking Strategy — Row-per-Document ✅ Chosen
Each CSV row becomes exactly one vector — no splitting
▼
Strategy Vectors Created Context Loss Storage Reason
✅ Row-per-document 15,420 None ~90MB Each row is already 150-200 tokens — perfectly sized. Splitting destroys claim context.
❌ Fixed-size (500 tokens) Would split rows Severe Same Designed for long docs — our rows are already <500 tokens. No benefit.
❌ Recursive chunking Wrong tool Severe 2-3× Designed for narrative text (PDFs, reports) — our data is structured key-value pairs
⚠ Enriched row (future) 15,420 None ~110MB Add computed fraud signals to each row before embedding — would improve search accuracy
🚀 Enriched Row + Hierarchical Store Same ~110MB 2 vectors/claim Store 2 vectors per claim: (1) short fraud-signal summary for fast search, (2) full enriched document for precise ranking — best of both speed and accuracy
